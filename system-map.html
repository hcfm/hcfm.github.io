The Weekly Report Workflow

Where I write it
My code editor
I build and fix the instructions in a program called VS Code. This is the writing side.
Where it runs
My computer, on its own
The computer runs those instructions automatically each week. I open the Terminal only to test by hand.

A "script" is just a small set of written instructions I created that does one job on its own, like "go get this week's website numbers." The report is a few of these running in order.

How does it run itself?

Your Mac has a built-in scheduler. I set a job that says "every Monday at 6am, run the report," and the Mac runs it start to finish, then goes quiet until the next week. It is the computer's own clock, not an outside website or cloud service. It only needs the Mac to be on or able to wake at that time.

Why a report gets "held," and what that protects

The gate chooses "nothing" over "wrong"

I built this because the biggest risk in reporting is sending a wrong or incomplete number to leadership and having them act on it. Before any report goes out it must pass a checklist I wrote. If any blocking check fails, the report is held and nothing is sent until I look at it. It protects the reporting against:

  • A data pull that failed or came back empty, so a section would be blank or misleading.
  • Numbers that do not reconcile, or a figure that cannot be verified against the platform.
  • Bot-inflated or non-human traffic slipping through into a headline number.
  • A week that is missing days of data, which would understate or overstate performance.
  • Any percentage shown without the real count behind it.

A held report is the system working correctly. It is far better to send nothing and fix it than to send something inaccurate that gets repeated upstairs.

The details that answer the rest

HubSpot: part of the system, waiting on its token

HubSpot is part of the reporting system. It just needs its API token to run, exactly the way Google Analytics and Search Console need read access. It measures who actually acted: email opens and clicks, form submissions, new contacts, and the Global Rosary sign-ups tied to our tracking codes. That is the step past reach, from "people saw it" to "people signed up," which is often what leadership most wants to know. Without it, the report can show that a reel reached millions but not how many of them registered.

What it needs: a read-only API token, a HubSpot "Private App," created by our HubSpot admin. Only an admin can issue it. The moment it is in place, HubSpot's numbers flow into the weekly report like every other source.

How it connects to each platform

Each platform needs its own read-only key or login for the automation to read it. Meta and YouTube are connected. Google Analytics and Search Console share one Google access, which still needs read access granted. HubSpot is part of the system too; it just needs its own read-only API token from our HubSpot admin, the same way Google Analytics needs read access. Every connection is read-only: it can pull numbers, never change, post, or configure anything.

If something goes wrong

Every run writes a log I can read. If a pull fails or a check does not pass, the run holds instead of shipping, and I see it in the log and fix the code. Nothing broken ever reaches the team silently.

How much it covers

Every week it pulls from 15+ website properties, 8 social accounts, and 4 YouTube channels, cleans it, and turns it into the reports, on its own. HubSpot (email and sign-ups) is the fifth source; it flows in the same way the moment its token is granted.
